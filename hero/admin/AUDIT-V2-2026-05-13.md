# Audit V2 banque d'assets — 2026-05-13

## CRITIQUE (à fixer avant test user)

- **`asset_usage.asset_id` + `section_timeline.asset_id` sans FK physique** — `082_assets_bank_v2.sql:103,123` : le commentaire dit « FK logique » mais aucune contrainte n'existe. Conséquences directes : (a) la cascade `ON DELETE` côté livres ne nettoie PAS les assets orphelins ; (b) on peut insérer un `asset_id` qui ne pointe vers rien (corruption silencieuse) ; (c) la migration 083 (cf doc) cite `ON DELETE CASCADE (book_id) — Postgres a besoin d'un index` qui existe bien, mais l'inverse (asset deleted → usage/timeline orphelin) n'est garanti que par le code TS de `DELETE /api/assets/[type]/[id]`. Si quelqu'un fait un `DELETE FROM assets_image` direct → orphans dans timeline + usage. **Fix** : ajouter triggers DB ou 4 FK conditionnelles (par type).
- **Aucune transaction côté DELETE asset** — `api/assets/[type]/[id]/route.ts:106-124` : 3 `await` séparés (`section_timeline` → `asset_usage` → asset). Si crash entre 2 et 3, l'asset reste mais tous les liens sont coupés (zombie en banque, invisible car asset_usage vidé). **Fix** : RPC plpgsql atomique `delete_asset(type, id)`.
- **POST `/api/assets/[type]` accepte n'importe quel champ** — `route.ts:108-117` : `assetFields = { ...body }` puis insert direct. Si le client envoie `id: 'attacker-uuid'` ou `created_at`, c'est inséré. Pour `assets_audio`, `kind` est NOT NULL CHECK IN ('sfx','music') mais aucune validation côté API → 500 cryptique au lieu de 400. **Fix** : whitelist par type des colonnes admissibles.
- **`addAnimationPellicule` au load V2 (AnimationStudioInner)** — `AnimationStudioInner.tsx:418` : à chaque mount, on `addAnimationPellicule(...)`. Si l'effet re-run (StrictMode dev, dep changes), on crée des doublons. Le contexte n'est pas vidé avant. **Fix** : `resetAnimationPellicules()` + `setAnimationPellicules([{...}])` ou guard par `pellicules.length===0`.
- **`new-layout` save Ctrl+S sur asset V2 → branche legacy** — `new-layout/page.tsx:1060,1122` : `savePlanToDb` ne reconnaît PAS `_assetId`. Le synthetic n'a pas `_planId`, donc le code tombe dans le MODE LEGACY `POST /api/sections/[id]/plans` avec `planIndex: 0`. Conséquence : tout save d'un asset image V2 écrase `section.images[0]` du JSONB legacy au lieu de PATCH `/api/assets/image/[id]`. Très grave (corruption silencieuse). **Fix** : ajouter branche `if (_assetId) PATCH /api/assets/image/[_assetId]`.

## HAUTE (UX bloquante)

- **Aucune vérification d'ownership** — toutes les routes API (`/api/assets/*`, `/api/sections/[id]/timeline`, `/api/asset-usage`) utilisent `supabaseAdmin` (service_role, bypass RLS) et NE VÉRIFIENT PAS que `bookId` appartient au user. En prod : n'importe qui appelant `POST /api/asset-usage { book_id: <livre d'un autre> }` peut polluer la library d'un autre auteur ou exfiltrer des assets via GET. **Fix** : auth middleware + check `books.user_id = current_user`.
- **`AnimationStudioInner` load V2 ne re-charge pas si crash GET** — `:355-360` : si `assetRes.ok` est false, on fait un fallback `fetch /api/sections/[id]` mais on continue d'appeler `assetRes.json()` (ligne 366) qui jettera (no body / wrong shape). Le `setError(msg)` ne se déclenche pas dans le bon chemin. **Fix** : early-return si !assetRes.ok.
- **Drag-drop library mapping `assetId` cassé** — `SectionTimelineEditorV2.tsx:132-135` : `payload.data.assetId ?? payload.data.imageId ?? payload.data.audioId ?? payload.data.pelliculeId`. OK pour images/audio mais pour `blockKind: 'video'`, `TimelineLibrary.tsx:176` ne met QUE `pelliculeId`+`shotId` ; or côté V2 `pelliculeId` = `LibraryAnimation.id` = asset_id (mappage dans `studio-section/page.tsx:112-113` `pelliculeId: a.id`). Marche par accident mais devient bug dès qu'on ajoute des shots vrais.
- **`previewPellicules` non typé conforme** — `studio-section/page.tsx:164-193` : on synthétise des pellicules avec 1 shot fake `shot:'medium', camera:'static'`, characterIds=[], speakerId=null. Si `AnimationStudioPreview` lit `pell.shots[i].cropKeyframes`/`textOverlays`/`perCharacter` pour faire le scrub bar ou afficher overlays, NaN/no-op silencieux. Idem `scene_visible`/`scene_offscreen`/`characters_appearance` toujours `null`.
- **Refresh complet après chaque mutation** — `studio-section/page.tsx:294,310,346,372` : `fetchAll(sectionId)` après chaque create/delete/resize/drop. 5 fetch concurrents (section + book + timeline + 3 banks) à chaque drag. Pour 1 timeline avec 30 blocs : ressenti laggy + flash de la library + perd la sélection. **Fix** : optimistic update local + refetch ciblé.
- **DELETE asset cascade trop large** — `api/assets/[type]/[id]/route.ts:106` : supprime `section_timeline` de TOUTES les sections de TOUS les livres référençant cet asset. Si un asset est partagé avec 3 livres et l'auteur de l'un d'eux clique poubelle → on détruit la timeline des 2 autres livres. **Fix** : interpréter "Supprimer" comme "retirer de MON livre" (DELETE asset_usage + DELETE section_timeline scopés bookId), garder asset si refs > 0.

## MOYENNE (risque latent)

- **`UNIQUE (asset_type, asset_id, book_id, section_id)` avec `section_id NULL`** — `082:107` : Postgres traite NULL comme non-égal en UNIQUE, donc on peut insérer N rows `(image, X, book, NULL)`. **Fix** : `UNIQUE NULLS NOT DISTINCT` (PG15+) ou expression index avec `COALESCE`.
- **Migration data non transactionnelle** — `migrate-section-images-to-assets.mjs:90-330` : N×M INSERT séquentiels (1 RTT chacun). Pour 1000 sections × 20 images, ≈ 100k RTT, ~30+ min. Si interruption au milieu, état partiel. **Fix** : batch INSERT (`.insert([...])`) par 100 ou wrap in `rpc('migrate_batch', ...)`.
- **GET timeline → 4 queries séquentielles** — `api/sections/[id]/timeline/route.ts:57-68` : 1 query par type d'asset. Postgres supporte parfaitement un `LEFT JOIN LATERAL` ou un view qui unifie ça. Réduit RTT.
- **Pas de cleanup orphans automatique** — `082` + tous les endpoints : assets sans refs (= 0 row dans `asset_usage`) ne sont jamais supprimés. Avec le bug critique ci-dessus (DELETE non-atomique), on accumule.
- **`POST /api/sections/[id]/timeline` `upsert asset_usage` sans contrôle book ownership** — `route.ts:116-121` : on récupère `section.book_id` puis upsert. Si l'auteur n'a pas accès au livre, on lui permet d'ajouter ses assets sur n'importe quelle section.
- **ImportFromBookModal absence de filtre par section** — comme tu l'as identifié. Pour un livre cible avec 200 assets, scroll infini. **Fix** : ajouter dropdown `sectionId` de filtrage + paramètre `sectionId` au GET.
- **Aucun index sur `assets_image.created_at`** — `083:135-137` : commenté « skip V1 », mais l'API fait `.order('created_at', { ascending: false })` (`api/assets/[type]/route.ts:76`) à CHAQUE fetch library. Sort en mémoire OK pour < 1k, à monitorer.
- **`(payload.blockKind === 'image_static')` mappé sur `image_static` mais SectionTimelineEditorV2 attend `image`** — `SectionTimelineEditorV2.tsx:142-146` : oui le mapping est correct (`image_static → image`), mais le commentaire de `TimelineBlock.kind` côté types autorise potentiellement `'image_static'` comme alias → double vérifier avant prod.
