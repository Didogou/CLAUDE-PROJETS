## A. Library désynchronisée

### A.1 [bug confirmé] folder Images de la library ne se met PAS à jour quand `section.images` change après auto-save
- **Fichier** : `src/app/editor-test/animation-studio/AnimationStudioInner.tsx:86,372-380`
- **Symptôme** : L'auteur drag-drop une image de la library Images → elle reste visible MAIS si l'auteur recharge ou que l'auto-save (PATCH `/api/sections/[id]`) écrase l'entrée `section.images[planIndex]` (kind `image` → kind `animation`), au prochain render l'image disparaît / le label change.
- **Cause** : `sectionBankImages` est un `useState` hydraté **une seule fois** au mount (useEffect avec deps `[planIndexFromUrl, sectionIdFromUrl, planId]`). Il n'écoute jamais les changements de `section.images` ultérieurs. De plus, `persistPlan` (lignes 626-696) force `images[planIndexFromUrl].kind = 'animation'` SANS vérifier le kind d'origine — donc si l'auteur a ouvert un plan kind='image' avec `/animation-studio` par erreur, l'entrée est mutée en kind='animation' sans avertissement.
- **Fix proposé** :
  1. Recomputer `sectionBankImages` à chaque auto-save réussi (réutiliser le résultat du fetch dans `persistPlan` plutôt qu'un re-fetch).
  2. Ne PAS forcer `kind: 'animation'` dans `persistPlan` si l'entrée existante était kind='image' — refuser ou prompter.

### A.2 [bug confirmé] Studio Section : folder Animations affiche AUSSI les pellicules déjà placées sur la timeline
- **Fichier** : `src/app/editor-test/animation-studio/components/multi-track-timeline/MultiTrackEditor.tsx:215-232`
- **Symptôme** : Dans `/studio-section`, la library Animations liste **toutes** les pellicules avec `videoUrl != null` du `animationPellicules` context. Or, dans Studio Section, ce contexte EST l'aplatissement de la timeline (= toutes les pellicules de la section). Donc chaque pellicule sur la timeline apparaît aussi en library. Tu n'as pas de banque "transverse au plan" — la library DOUBLE la timeline.
- **Cause** : `libraryAnimations = animationPellicules.filter(p => p.videoUrl != null)` sans dédup contre `state.blocks`.
- **Fix proposé** : Calculer `placedPelliculeIds = new Set(state.blocks.filter(b => b.kind === 'video' || b.kind === 'image_static').map(b => b.pelliculeId))` et `libraryAnimations = animationPellicules.filter(p => p.videoUrl && !placedPelliculeIds.has(p.id))`. Idem pour `libraryImages`.

### A.3 [bug] `pelliculeIdToPlanIndexRef` reste stale entre 2 renders
- **Fichier** : `src/app/editor-test/studio-section/page.tsx:217, 270-274`
- **Symptôme** : Si l'auteur clique sur un bloc juste après une suppression mais avant le re-render React (très improbable mais possible avec batch d'events), le routing utilise l'ancien planIndex → ouvre la mauvaise pellicule.
- **Cause** : La ref n'est mise à jour que dans `useEffect`, donc 1 frame après le set. Pas critique en pratique. Pourrait être un `useMemo` à la place.

### A.4 [bug] Studio Section : aucune AUTO-SAVE → toutes modifs timeline perdues
- **Fichier** : `src/app/editor-test/studio-section/page.tsx` (aucun auto-save), `src/components/studio-section/SectionTimelineEditor.tsx` (aucun)
- **Symptôme** : Resize d'un bloc audio, drag-drop SFX/musique/texte, déplacement d'un overlay → ça mute `animationPellicules` via le mapper inverse → mais **rien ne PATCH `/api/sections/[id]`**. À la navigation, tout est perdu.
- **Cause** : Studio Section a été conçu en lecture-seule pour l'orchestration, pas pour l'édition. Mais on a câblé `MultiTrackEditor` qui mute le state.
- **Fix proposé** : Ajouter un debounce auto-save identique à AnimationStudioInner (lignes 698-710) qui sérialise `animationPellicules` → reconstruit `section.images` via le mapper inverse + PATCH `/api/sections/[id]`. Ou désactiver les mutations du MultiTrackEditor en Studio Section tant qu'il n'y a pas d'auto-save.

## B. Suppression DB + Storage

### B.1 [bug confirmé] Pellicules image_static (synthétiques) NON supprimables depuis Studio Section
- **Fichier** : `src/app/editor-test/studio-section/page.tsx:337-398`
- **Symptôme** : Cliquer corbeille sur un bloc image fixe dans la timeline (synthétique id `__ss-static-N` ou `__ss-empty-anim-N`) → `console.warn('[handleDeleteAnimation] plan sans pellicules')` ou `'pellicule introuvable dans plan'` → la pellicule reste affichée. Aucun feedback UX.
- **Cause** : `handleDeleteAnimation` n'accepte que les pellicules réelles avec `plan.pellicules.find(...)`. Les ids synthétiques `__ss-static-X`, `__ss-empty-anim-X`, `__ss-choice-X` (cf `flattenSectionImages`) ne matchent jamais.
- **Fix proposé** :
  - Si `pelliculeId.startsWith('__ss-static-')` → suppression d'une entrée kind='image' du `section.images` (= splice complet + delete storage de l'URL).
  - Si `pelliculeId.startsWith('__ss-empty-anim-')` → splice de l'entrée animation vide.
  - Si `pelliculeId.startsWith('__ss-choice-')` → splice de l'entrée kind='choice' + delete `choice_data.image_url`.

### B.2 [bug confirmé] `onDeleteImage` JAMAIS câblé → corbeille images library invisible
- **Fichier** : `src/app/editor-test/animation-studio/components/multi-track-timeline/MultiTrackEditor.tsx:64,77,260` et `TimelineLibrary.tsx:95,203` ; `src/components/studio-section/SectionTimelineEditor.tsx` (pas wiré)
- **Symptôme** : Dans la library Images, il n'y a aucun bouton corbeille. L'auteur ne peut pas supprimer une image de la banque depuis le Studio.
- **Cause** : Le prop `onDeleteImage` est défini dans la signature mais aucun caller ne le fournit. `<DragItem ... onDelete={onDeleteImage ? () => onDeleteImage(i.id) : undefined}>` → undefined → bouton non rendu.
- **Fix proposé** : Câbler `onDeleteImage` au niveau Studio Section (suppression de `section.images[planIdx]` kind='image' + DELETE storage de `images[planIdx].url`).

### B.3 [bug] `audioTracks[].audioUrl` JAMAIS supprimés du Storage à la suppression d'une pellicule
- **Fichier** : `src/app/editor-test/studio-section/page.tsx:358-362` ; `src/app/editor-test/animation-studio/AnimationStudioInner.tsx:1359-1366`
- **Symptôme** : Quand on supprime une pellicule qui a des audioTracks attachés, les `audioUrl` (sfx/music) restent orphelins dans le bucket `audio`.
- **Cause** : `urlsToDelete = [videoUrl, firstFrameUrl, lastFrameUrl]` — pas d'inclusion de `p.audioTracks?.map(a => a.audioUrl)`.
- **Note** : Si l'audio est dans la banque livre, le supprimer du Storage casserait les autres pellicules qui le réutilisent → c'est OK de ne PAS le supprimer ICI. Mais alors il faut prévoir une corbeille banque audio séparée (qui DELETE storage + check refs cross-pellicules).
- **Fix proposé** : Le laisser pour la corbeille banque audio dédiée (Phase 2 timeline). À documenter explicitement dans handleDeleteAnimation pour qu'on ne croit pas avoir le cas couvert.

### B.4 [bug] `splice(planIdx, 1)` quand plan vide → DESYNC `pelliculeIdToPlanIndexRef`
- **Fichier** : `src/app/editor-test/studio-section/page.tsx:381`
- **Symptôme** : Si l'auteur supprime la dernière pellicule d'un plan animation, `splice` retire l'entrée complète, ce qui décale les planIndex des entries suivantes. Avant le re-render, la map est stale. Click rapide sur une pellicule suivante → ouvre la mauvaise.
- **Cause** : Mutation par index sans recalcul synchrone de la map.
- **Fix proposé** : Recalculer la map IMMÉDIATEMENT après le splice (avant `setSectionImages`), ou utiliser des planIds stables au lieu d'indexes.

### B.5 [bug] Race auto-save : `read-then-write` sur `section.images` sans optimistic concurrency
- **Fichier** : `src/app/editor-test/animation-studio/AnimationStudioInner.tsx:626-696` (persistPlan) ; `src/app/api/sections/[id]/route.ts:15` (PATCH full replace)
- **Symptôme** : Si 2 onglets (ou animation-studio + studio-section) PATCH `section.images` en même temps, le 2nd écrase les modifs du 1er. Pas de version-check, pas de merge.
- **Fix proposé** : Soit ajouter une `version` / `updated_at` check optimiste, soit faire un PATCH ciblé par `planIndex` (= side-effect-only sur cette entry, le reste préservé côté serveur).

### B.6 [bug] Champ `kind: 'animation'` forcé dans `persistPlan` même si entry était kind='image'
- **Fichier** : `src/app/editor-test/animation-studio/AnimationStudioInner.tsx:650-658`
- **Symptôme** : Si l'auteur a opened `/animation-studio?sectionId=X&planIndex=N` sur une entry initialement kind='image' (incohérence URL), le 1er auto-save mute l'entry en kind='animation', perdant les autres champs (tags, conditions, etc.).
- **Fix proposé** : Vérifier `existing.kind === 'animation'` avant de PATCH. Sinon, throw / no-op + log.

### B.7 [bug] DELETE Storage `await fetch` AVANT update DB → si DELETE échoue, on quand même PATCH (orphans inversés)
- **Fichier** : `src/app/editor-test/studio-section/page.tsx:363-389`
- **Symptôme** : Si `/api/storage/delete` plante (500), on log un warn mais on PATCH quand même. Au prochain reload, les pellicules sont absentes de la DB mais leurs fichiers Storage persistent → orphans Storage.
- **Cause** : `try { await fetch(delete) } catch (err) { console.warn(...) }` → continue.
- **Note** : Le commentaire dit "non-bloquant, l'auteur veut surtout retirer de la timeline". OK conceptuellement mais devrait être visible quelque part pour qu'un cron de nettoyage puisse rattraper.
- **Fix proposé** : Logguer le orphan dans une table `storage_orphans` ou simplement noter que la DELETE Storage est best-effort dans la doc.
